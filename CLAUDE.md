# Agentic Usage ‚Äî Splunk Dashboard App

This repo contains the `agentic_usage` Splunk app: vendor-neutral team
observability dashboards for agentic coding harnesses (Claude Code, Claude
Desktop, Cowork, Gemini CLI, OpenAI Codex CLI), built entirely from custom
visualizations, plus the Claude Code skill used to develop them. Telemetry is
normalized upstream by the sibling `ai-harness-otel` collector config; the
dashboards query only the normalized contract. Three dashboards: Overview,
Sessions, and Data Health (the setup/ingest-health screen; Overview/Sessions
carry a self-hiding `cu_checklist` banner that links to it while no data
arrives).

## Key Paths

- **App**: `agentic_usage/` ‚Äî the deployable Splunk app (dashboard XML, conf files, built viz bundles)
- **Viz sources**: `examples/cu_*/` ‚Äî each subdirectory is a standalone Splunk viz app (source of truth)
- **Shared CSS**: `shared/agentic_usage_fonts.css` (embedded fonts), `shared/agentic_usage_theme.css` (light-theme overrides)
- **Build**: `./agentic_usage/build.sh` ‚Äî builds all cu_* vizs, merges them into the app, packages `dist/agentic_usage.tar.gz`
- **Collector overlay**: `otel/splunk.yaml` ‚Äî Splunk-specific exporter overlay composed onto `ai-harness-otel/config/collector.yaml` (the normalization logic lives in that repo, NOT here)
- **Skill**: `.claude/skills/splunk-viz/SKILL.md` ‚Äî the full skill definition for building Splunk custom visualizations
- **Test harness**: `test-harness.html` + `harness-manifest.json` ‚Äî browser-based viz testing (see TEST-HARNESS.md)

## Rules

- **Dashboards query only the normalized telemetry contract** ‚Äî macros
  `agentic_metrics`/`agentic_data`, additive accounting metric
  `agentic.token.usage` (gated by `agentic.token.relationship`), other
  `agentic.*` metrics/events, and cataloged `gen_ai.*` attributes (see
  ai-harness-otel/ATTRIBUTE_REFERENCE.md). Never reference harness-native names
  (`claude_code.*`, `gemini_cli.*`, `codex.*`) or uncataloged pass-through
  fields in dashboard SPL. **Sanctioned exceptions:** the Overview's
  `ds_drift` Contract-drift panel deliberately probes the drift signatures
  NORMALIZATION.md tells consumers to query for (surviving raw source keys,
  wildcard-suffixed renamed events ‚Äî exempted via `PROBE_EXEMPT` in the
  validator), and the Data Health page + `ds_banner` probes query `otelcol_*`
  collector self-metrics, wildcard `agentic.*` names, and index presence to
  diagnose ingest. Do not extend either pattern to any other search.
- Run `ruby scripts/validate-dashboard-contract.rb` (pass the path to
  ai-harness-otel/ATTRIBUTE_REFERENCE.md, or set
  `AGENTIC_ATTRIBUTE_REFERENCE`) before committing any dashboard SPL change ‚Äî
  it gates contract-token cataloging, token-relationship gating, identity
  fillnull sentinels, `[| mstats` subsearches, and pre-normalization field
  names across all views.
- Harness filtering/grouping goes through `harness_key(2)` (well-known
  `service.name` first, `agentic.harness.name` fallback for custom
  `OTEL_SERVICE_NAME`, raw name last) and labels through `harness_label(1)` ‚Äî
  never compare `service.name` to the Harness dropdown token directly.
- Visualization source lives in `examples/{viz_name}/` ‚Äî never edit the merged copies under `agentic_usage/appserver/` directly; they are regenerated by the build
- Follow the directory structure defined in the skill (do not deviate)
- Use ES5 syntax only in visualization_source.js (var, function, for ‚Äî no const/let/arrow)
- Always handle HiDPI, null ctx, and zero-size canvas
- Keep `visualization.css` background transparent by default
- Every setting in `formatter.html` must also appear in `savedsearches.conf.spec`
- JS defaults must match formatter HTML defaults (Splunk doesn't send formatter defaults on first load)
- Never read `config` in `formatData` ‚Äî do it in `updateView`
- 100% custom vizs (DOM or Canvas) ‚Äî never native Splunk panels that break the typography; the app uses HYBRID (DOM text + Canvas charts)
- Use `./agentic_usage/build.sh` to build & package ‚Äî it rebuilds each viz with webpack, rewrites the property namespace to `agentic_usage.{viz}`, merges conf files, prepends fonts/theme CSS, and bumps the app version
- New harness onboarded in the collector? App-side change is only: `harness_key(2)` + `harness_label(1)` macros + the Harness dropdown items in both dashboards (+ context-window lookup rows if it brings new models). Data Health needs no edit ‚Äî its per-harness board derives rows from `harness_key(2)`
- New dashboard? Add it to the `DASHBOARDS` registry in `agentic_usage/build.sh` ‚Äî the build regenerates `nav/default.xml` from that array and would otherwise drop the view from the nav
